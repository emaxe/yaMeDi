# AI Agent Guidelines (AGENTS.md)

Этот файл содержит инструкции и лучшие практики для AI агентов и разработчиков, работающих с данным проектом. Пожалуйста, соблюдайте эти правила при генерации кода, рефакторинге и добавлении новых функций.

## 1. Стек технологий
- **Frontend:** React 18, TypeScript, Vite, Tailwind CSS, Recharts
- **Desktop (Wrapper):** Electron (main + preload)
- **Сборка:** electron-builder
- **Форматирование и линтинг:** ESLint, Prettier
- **Тестирование:** Vitest (unit), Playwright (E2E)

## 2. Архитектурные правила
- **Разделение слоев:** 
  - Компоненты UI должны быть максимально "глупыми" (dumb components) там, где это возможно.
  - Сетевые запросы к API (Yandex Metrica, Yandex Direct, UON) должны находиться в директории `src/api/`.
  - Сложная бизнес-логика (расчет KPI, аггрегация данных, экспорт) выносится в `src/lib/`.
- **Типизация (TypeScript):** 
  - Избегайте использования `any`.
  - Все интерфейсы, типы API ответов и внутренние модели данных должны быть описаны в `src/types/`.
- **Состояние (State):** 
  - Используйте React Hooks (`useState`, `useMemo`, `useEffect`) для локального состояния.
  - Для глобального состояния и работы с данными API используйте хуки инкапсуляции (например, `src/hooks/useKpiData.ts`, `src/hooks/useUonLeads.ts`).

## 3. Правила написания кода
- **Именование:**
  - Компоненты: `PascalCase.tsx` (например, `DashboardWidget.tsx`).
  - Утилиты, хуки, API-клиенты: `camelCase.ts` (например, `useKpiData.ts`, `kpi-utils.ts`).
  - Интерфейсы: `IPascalCase` или просто `PascalCase` в зависимости от контекста.
- **Стилизация:**
  - Используйте Tailwind CSS через проп `className`.
  - Избегайте написания кастомного CSS, если задачу можно решить утилитами Tailwind.
  - Для объединения классов используйте `clsx` или `tailwind-merge` (функция `cn` в `src/lib/utils.ts` при наличии).

## 4. Специфика Electron
- **Безопасность:** Рендерер-процесс (React) **не должен** иметь прямого доступа к API Node.js (fs, child_process и т.д.).
- **IPC (Inter-Process Communication):** Все взаимодействия с ОС (чтение файлов, вызов `safeStorage`) должны происходить исключительно через IPC мост, объявленный в `electron/preload.ts` и обрабатываемый в `electron/main.ts`.
- **Токены:** OAuth-токены и другие чувствительные данные сохраняются безопасно с использованием `safeStorage` в Electron.

## 5. Модули и функции (Недавние обновления)
- **Экспорт данных:** Логика формирования отчетов (PDF, XLSX) должна оставаться в `src/lib/pdfExport.ts` и `src/lib/xlsxExport.ts`.
- **Интеграции:** При добавлении новых API (например, amoCRM, Bitrix) создавайте отдельный модуль в `src/api/` по аналогии с `src/api/uon.ts`.

## 6. Тестирование
- Каждый новый модуль логики (`src/lib/`, `src/api/`) должен покрываться unit-тестами с использованием Vitest (`*.test.ts`).
- Для сложных компонентов UI пишите тесты рендеринга (`*.test.tsx`).
- Не ломайте существующие тесты при рефакторинге. Перед коммитом всегда запускайте `npm run test:unit`.
